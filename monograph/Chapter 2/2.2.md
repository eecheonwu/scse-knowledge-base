# 2.2 Defining the Single Source of Truth

The concept of a **Single Source of Truth (SSOT)** has been widely adopted across information systems, data management, enterprise architecture, and software engineering. Organizations have long recognized the importance of maintaining an authoritative source for business information in order to reduce redundancy, improve consistency, and establish confidence in decision-making. As a result, the term SSOT has become synonymous with centralized repositories, master data management systems, configuration databases, and documentation platforms that provide trusted information to multiple stakeholders.

Although these interpretations have proven valuable for managing information assets, they are insufficient for addressing the broader challenges of modern software engineering. Contemporary software systems are no longer constrained by the availability of information alone; rather, they are constrained by the ability to preserve, interpret, govern, and continuously evolve engineering knowledge. This distinction marks a fundamental departure from traditional understandings of the Single Source of Truth and forms the conceptual basis of **SSOT-Centric Software Engineering (SCSE)**.

## 2.2.1 Traditional Interpretations of the Single Source of Truth

Historically, the Single Source of Truth has been understood primarily as an authoritative repository of information. Within enterprise systems, an SSOT typically ensures that business entities such as customers, products, financial records, or inventory data are maintained consistently across multiple applications. Rather than allowing duplicate or conflicting records to exist within different databases, organizations designate a single authoritative system from which all other systems derive their information.

Similarly, software engineering organizations frequently establish centralized repositories for source code, requirements documentation, architecture diagrams, configuration files, or project documentation. Version control systems such as Git provide an authoritative history of source code. Requirements management tools preserve stakeholder requirements. Architecture repositories document system designs, while infrastructure management systems maintain operational configurations.

The primary objective of these implementations is consistency. By reducing duplication and ensuring that information originates from an authoritative location, organizations improve data quality, simplify governance, and reduce operational errors.

However, consistency of information does not necessarily imply consistency of engineering knowledge.

A requirements repository may accurately capture stakeholder needs while remaining disconnected from architectural decisions. Architecture documentation may faithfully describe system structure without preserving the rationale behind design choices. Source code may correctly implement functionality while providing little insight into business intent or future evolution. Each repository may individually function as a “single source of truth” for a particular artifact, yet the engineering knowledge connecting these artifacts remains fragmented.

Consequently, traditional SSOT implementations often produce **multiple isolated sources of truth**, each authoritative within its own domain but collectively incapable of representing the complete engineering understanding of the software system.

## 2.2.2 Information Is Not Engineering Knowledge

A central proposition of this book is that **information should not be confused with engineering knowledge**.

Information consists of facts, records, specifications, diagrams, source code, documents, configuration files, and other observable engineering artifacts. Information answers questions such as:

·        What is the system requirement?

·        Which API endpoint exists?

·        Which database table stores customer information?

·        Which service implements payment processing?

Engineering knowledge extends beyond these facts.

Engineering knowledge explains:

·        Why was this requirement introduced?

·        Why was this architectural pattern selected?

·        Which assumptions influenced this implementation?

·        Which constraints must always be preserved?

·        Which components are semantically related?

·        What consequences will result from changing a particular feature?

Information describes the system.

Engineering knowledge explains the system.

This distinction is particularly significant because software engineering decisions rarely depend upon isolated information. Engineers reason using relationships among information, historical context, architectural rationale, domain understanding, and accumulated experience. It is these relationships that enable informed decision-making during software evolution.

An engineering Single Source of Truth must therefore preserve not only engineering information but also the semantic relationships that transform information into engineering knowledge.

## 2.2.3 Limitations of Existing SSOT Implementations

While traditional SSOT implementations successfully eliminate redundant information, they generally exhibit several limitations when viewed from the perspective of software engineering.

First, they are **artifact-centric** rather than **knowledge-centric**.

Most repositories are organized around specific engineering artifacts:

·        source code repositories,

·        requirements repositories,

·        documentation repositories,

·        architecture repositories,

·        deployment repositories.

Each repository governs one category of engineering output but rarely maintains semantic relationships with the others.

Second, they are largely **static**.

Although version control systems preserve historical changes, they typically record modifications to artifacts rather than changes to engineering intent. They answer **what changed**, but not necessarily **why it changed** or **how the change affects the engineering knowledge of the overall system**.

Third, they provide limited support for **knowledge continuity**.

As software systems evolve, engineering decisions accumulate over years or even decades. Existing SSOT implementations seldom preserve the complete reasoning process behind those decisions. Consequently, organizations repeatedly rediscover architectural knowledge that once existed but was never formally represented.

Fourth, existing SSOT implementations provide minimal support for **machine reasoning**.

Large Language Models, autonomous engineering agents, and intelligent planning systems require explicit representations of concepts, relationships, constraints, dependencies, and engineering rationale. Conventional repositories primarily store documents and source code, leaving intelligent systems responsible for reconstructing context that was never explicitly modeled.

These limitations suggest that existing interpretations of SSOT address information governance but not engineering knowledge governance.

## 2.2.4 An Engineering Interpretation of the Single Source of Truth

Within **SSOT-Centric Software Engineering**, the Single Source of Truth is redefined as the authoritative representation of engineering knowledge rather than merely the authoritative storage of engineering information.

**Definition 2.1 (Engineering Single Source of Truth).**

**An Engineering Single Source of Truth (Engineering SSOT) is the authoritative, continuously evolving, semantically structured representation of engineering knowledge from which all software engineering artifacts derive their meaning, consistency, traceability, and intent throughout the software lifecycle.**

This definition differs fundamentally from traditional interpretations in several important respects.

First, the Engineering SSOT governs **knowledge**, not merely information.

Second, it preserves **engineering intent**, not simply engineering artifacts.

Third, it maintains **semantic relationships** among engineering concepts rather than storing isolated documents.

Fourth, it supports **continuous evolution**, ensuring that engineering knowledge remains synchronized with every software artifact throughout the lifecycle.

Finally, it serves both **human engineers** and **intelligent software agents** by providing a common foundation for engineering reasoning.

Within this interpretation, source code is no longer regarded as the definitive representation of the software system. Instead, source code becomes one realization of the engineering knowledge preserved within the Engineering SSOT.

## 2.2.5 Characteristics of an Engineering SSOT

An Engineering Single Source of Truth possesses several defining characteristics that distinguish it from conventional repositories.

**Authoritativeness.** The Engineering SSOT serves as the definitive representation of engineering knowledge. All software artifacts must remain consistent with it, and engineering decisions derive their authority from its contents.

**Semantic Richness.** Knowledge is represented together with its relationships, dependencies, assumptions, rationale, and constraints rather than as isolated pieces of information.

**Traceability.** Every engineering artifact can be traced back to the engineering knowledge from which it originated, enabling complete lifecycle traceability.

**Knowledge Continuity.** Engineering knowledge evolves continuously without losing historical context or architectural intent.

**Machine Interpretability.** Knowledge is represented in forms that support computational reasoning by AI systems while remaining understandable to human engineers.

**Consistency Preservation.** Changes introduced within one engineering artifact propagate through the Engineering SSOT, allowing related artifacts to remain semantically synchronized throughout software evolution.

These characteristics collectively transform the SSOT from a passive repository into an active engineering knowledge system.

## 2.2.6 The Engineering SSOT as the Foundation of SCSE

The Engineering Single Source of Truth constitutes the cornerstone of **SSOT-Centric Software Engineering**.

Unlike conventional software engineering practices, where requirements, architecture, implementation, testing, deployment, and operations are frequently managed through partially independent processes, SCSE organizes every engineering activity around the Engineering SSOT.

Requirements become structured representations of stakeholder knowledge.

Architecture becomes structured representations of system knowledge.

Implementation plans become structured representations of realization knowledge.

Source code becomes executable representations of implementation knowledge.

Tests become executable representations of validation knowledge.

Deployment pipelines become operational representations of infrastructure knowledge.

Operational monitoring becomes continuous representations of runtime knowledge.

Each artifact represents a different view of the same underlying engineering knowledge.

Consequently, software evolution becomes the controlled evolution of engineering knowledge rather than the independent modification of disconnected engineering artifacts.

## 2.2.7 Beyond Documentation Repositories

One of the most significant implications of this definition is that an Engineering SSOT should not be interpreted as another documentation platform.

A documentation repository stores documents.

A source code repository stores code.

A requirements management system stores requirements.

An Engineering SSOT governs the engineering knowledge that connects them all.

Documentation therefore becomes an output of the Engineering SSOT rather than its primary purpose. Architecture diagrams, implementation plans, test plans, deployment guides, operational procedures, and even source code may all be generated, validated, synchronized, or evaluated against the authoritative engineering knowledge maintained within the Engineering SSOT.

This distinction represents one of the principal contributions of this book. It shifts the concept of the Single Source of Truth from **information management** to **engineering knowledge management**, thereby establishing the conceptual foundation for a new software engineering paradigm.

The following section builds upon this definition by explaining why this book deliberately adopts the term **SSOT-Centric Software Engineering** rather than **SSOT-Driven Software Engineering**. This distinction is more than terminology; it establishes the hierarchical relationship between the overarching engineering paradigm and the methodologies that operate within it, providing the conceptual structure upon which the remainder of the SCSE framework is constructed.
